# 02. 아키텍처 (구현)

## 1. 기술 스택

- **Next.js 15 (App Router)** + TypeScript
- **Supabase**: Auth(이메일/비밀번호), Postgres, RLS
- **@supabase/ssr**: 서버/클라이언트/미들웨어 세션 처리
- **Tailwind CSS** 스타일링
- **Vercel** 배포

## 2. 폴더 구조

```
src/
├─ middleware.ts              # 인증 가드 + 매직링크 code 폴백
├─ types/database.ts          # 테이블 TS 타입
├─ lib/
│  ├─ booking.ts              # 슬롯 설정/계산 (공용 순수 함수)
│  ├─ utils.ts                # 날짜 포맷, 상태 라벨
│  └─ supabase/
│     ├─ client.ts            # 브라우저 클라이언트 (anon)
│     ├─ server.ts            # 서버 클라이언트 (anon, 쿠키 세션)
│     └─ admin.ts             # ★ service role (서버 전용, RLS 우회)
├─ components/
│  ├─ ScheduleGrid.tsx        # 주간 시간표 (예약 화면/공개 공용 가능)
│  └─ MonthSchedule.tsx       # 공개 월간 캘린더
└─ app/
   ├─ (고객/공개 페이지)
   ├─ admin/                  # 관리자 (layout에서 role 가드)
   └─ api/                    # 서버 라우트 (아래)
```

## 3. Supabase 클라이언트 3종 (중요)

| 클라이언트 | 키 | 사용처 | RLS |
|-----------|----|--------|-----|
| `client.ts` | anon | 'use client' 컴포넌트 | 적용됨 |
| `server.ts` | anon | 서버 컴포넌트/대부분 API (로그인 사용자 기준) | 적용됨 |
| `admin.ts` | **service role** | 일부 서버 API 전용 | **우회** |

- `admin.ts`는 `import 'server-only'`로 보호 → 클라이언트 번들에 들어가면 빌드 실패.
- service role은 **권한 검증을 코드로 직접 한 뒤에만** 사용한다.

### admin(service role)을 쓰는 곳과 이유
| 위치 | 이유 |
|------|------|
| `/api/availability` | 모든 고객 예약의 *시각만* 모아 빈 시간 계산 (RLS면 남의 예약을 못 봄) |
| `/api/appointments` (생성) | 중복 예약 충돌 검사 (남의 예약과 겹치는지 시각만 확인) |
| `/api/appointments/[id]/cancel` | 소유권·상태 검증 후 제한된 취소만 수행 (고객 update RLS를 안 열기 위함) |
| `/api/public-schedule` | 비로그인에게 승인 예약 시각만 반환 |

## 4. API 라우트

### 인증
- `GET /api/auth/callback` — 이메일 인증 콜백. `token_hash`(verifyOtp) + `code`(PKCE) 둘 다 처리. 프로필 없으면 `/onboarding`.
- `POST /api/auth/signout` — 로그아웃, **303**으로 `/login` (307이면 405)

### 프로필
- `POST /api/profile/create` — 최초 생성. 서버가 auth user 기준으로 `id/email` 결정, `role='customer'`, `privacy_agreed_at`=서버시각 강제.
- `PATCH /api/profile/update` — `name`/`phone`만 변경.

### 예약
- `POST /api/appointments` — 생성. 서버에서 프로필 스냅샷 구성 + 슬롯 유효성 + 과거 + **구간 겹침** 검증.
- `PATCH /api/appointments/[id]/cancel` — 본인·요청/승인 상태만 취소.
- `GET /api/availability?date=|from=&days=` — 잡힌 슬롯의 `{at, status, duration}`만 반환(개인정보 X).
- `GET /api/public-schedule` — 승인 예약의 `{scheduled_at, duration, label}`만.

### 문의 / 관리자
- `POST /api/contact` — 생성(서버 스냅샷, status=open 강제).
- `PATCH /api/admin/appointments/[id]` — 상태 변경(+타임스탬프).
- `PATCH /api/admin/contact/[id]` — 답변/상태(+answered_at).
- `POST,DELETE /api/admin/customer-notes[/[id]]` — 내부 메모 CRUD (admin 전용).

## 5. 인증 흐름

- **로그인**: 브라우저 `signInWithPassword` → @supabase/ssr가 세션 쿠키 기록 → 서버가 읽음.
- **회원가입**: `signUp` → 인증 메일 → 링크 클릭 → `/api/auth/callback` → 세션.
- **미들웨어**(`middleware.ts`): `/my`, `/admin` 비로그인 차단. `?code=`가 루트로 떨어지면 콜백으로 포워딩.
- **관리자 가드**: `app/admin/layout.tsx`에서 `profiles.role` 확인 후 아니면 `/my`.

## 6. 예약 슬롯 계산 (`lib/booking.ts`)

순수 함수라 클라이언트/서버 공용:
- `slotsForDate(dateStr)` — 그 날의 슬롯 목록(KST `+09:00` ISO로 생성 → 타임존 안전)
- `coveredInstants(iso, duration)` — 시작+소요시간이 차지하는 슬롯들
- `isValidBooking(iso, duration)` — 영업시간/슬롯배수/최대길이 검증
- `intervalsOverlap(...)` — 구간 겹침 판정

## 7. 주요 설계 결정

- **FK 제약 없음**: `customer_id` 등은 값만 보관, 무결성은 앱+RLS로. 런타임 FK 에러 방지.
- **연락처 스냅샷**: 예약/문의에 당시 email/phone을 복제 저장 → 프로필이 바뀌어도 당시 기록 보존.
- **타임존**: 모든 슬롯을 `+09:00` ISO 문자열로 만들어 서버(UTC)에서도 동일 결과.
- **클라이언트 입력 불신**: customer_id/email/phone/status는 서버가 재구성·강제.
- **개인정보 표면 최소화**: 공개/가용성 API는 시각·상태만 반환.
