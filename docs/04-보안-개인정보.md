# 04. 보안 · 개인정보

## 1. 신뢰 경계 (가장 중요)

**클라이언트가 보낸 값은 신뢰하지 않는다.** 모든 민감 결정은 서버가 한다.

| 항목 | 잘못된 방식 | 이 프로젝트의 방식 |
|------|------------|-------------------|
| customer_id | 클라이언트가 보낸 값 사용 | 서버가 `auth.uid()` 기준으로 강제 |
| customer_email/phone | 폼 입력값 저장 | 서버가 profiles에서 조회해 스냅샷 |
| role | 가입 시 클라이언트 지정 | 서버가 항상 `customer` |
| privacy_agreed_at | 클라이언트 시각 | 서버 시각 |
| status | 클라이언트 지정 | 생성 시 requested/open 강제 |
| admin_reply/answered_at | 고객이 수정 | 관리자 API에서만 |

검증됨: 위조 customer_id/email로 예약 생성 시도 → 서버가 무시하고 본인 값 사용 (E2E 통과).

## 2. 권한 (RLS + 코드)

- 1차 방어: **RLS** (DB가 행 단위 차단) — `03-데이터베이스.md`.
- 2차 방어: **서버 API 권한 체크** — 관리자 API는 `profiles.role==='admin'` 확인.
- `is_admin()`은 `security definer`로 RLS 재귀 없이 동작.

## 3. service role 키 취급

- `SUPABASE_SERVICE_ROLE_KEY`는 **서버 전용**. RLS를 우회하는 마스터 키.
- `NEXT_PUBLIC_` 접두사 금지. 'use client' 파일에서 import 금지.
- `src/lib/supabase/admin.ts`는 `import 'server-only'`로 보호(클라이언트 번들에 들어가면 빌드 실패).
- 사용 시 **반드시 권한·소유권을 코드로 먼저 검증**한 후에만.

## 4. 개인정보 노출 최소화

- **공개 스케줄 / 가용성 API**: `scheduled_at`(+duration/status/label)만 반환.
  `customer_id/email/phone/message/admin_message`는 **절대 미포함** (E2E로 검증).
- `appointments`에 anon select RLS 정책 **없음**. 공개 데이터는 서버 API로만.
- 공개 월간 캘린더: 승인 예약의 "시간"만, 누가 했는지 모름.
- 예약 화면(로그인): 예약중/승인 슬롯이 회색·색상으로만 보이고 고객 정보 없음.

## 5. 한국 개인정보보호법 대응

- 회원가입/온보딩에 **개인정보 처리 안내 + 기준일** 표시, 동의 체크 전 저장 버튼 비활성.
- 이메일·이름 = 필수, 휴대폰 = 선택(미입력 시 null, 기본 기능 사용 가능).
- 예약 요청사항/문의/메모 입력창에 **민감정보 입력 금지 안내**.
- `/privacy` 처리방침 페이지(상단 `OPERATOR_INFO` 상수만 고치면 됨).
- 처리방침 버전을 DB 컬럼/상수로 관리하지 않고 **기준일 텍스트**로만.

## 6. 인증 보안

- 로그인: 이메일+비밀번호(Supabase가 해시 저장, 앱은 비밀번호 미보관).
- 가입: 이메일 인증 1회(`enable_confirmations=true`) → 인증 전 로그인 차단(검증됨).
- 콜백은 `token_hash`/`code` 모두 처리. 로그아웃은 303 리다이렉트.

## 7. 비밀/배포 위생

- `.env.local`, `.vercel`, DB 비밀번호 파일은 `.gitignore`로 제외.
- Vercel 환경변수에서 service role 키는 서버 전용으로 등록.
- 로그에 이메일/전화/문의 본문/메모를 그대로 출력하지 않음.

## 8. 한계 / 주의

- 무료 플랜 기본 이메일은 발송량이 적음 → 운영은 SMTP 필요(`06`).
- 기본 인증 메일 템플릿은 PKCE(code) 방식 → 가입 확인은 **같은 브라우저**에서 권장.
  SMTP 연결 시 token_hash 템플릿으로 교체 가능(브라우저 무관).
- 이 템플릿은 법률 자문이 아님. 운영 전 처리방침·동의 문구 전문가 검토 권장.
