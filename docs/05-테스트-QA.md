# 05. 테스트 · QA

## 1. 원칙

- MVP이므로 무거운 테스트 프레임워크는 상주시키지 않음.
- 개발 중에는 **일회용 E2E 스크립트**(Playwright/supabase-js)로 핵심 플로우를 검증하고, 끝나면 삭제.
- 권한·RLS·개인정보 노출 방지·예약/문의 핵심 플로우는 반드시 검증.

## 2. 지금까지 자동 검증한 항목 (통과 기록)

### 핵심 플로우 E2E (32/32)
- 비로그인 `/my`·`/admin` → `/login` 리다이렉트
- 세션 주입 후 온보딩 → 프로필 생성(role=customer, phone null, privacy 서버시각, email 스냅샷)
- 예약/문의 생성 시 status 강제(requested/open) + customer_id/email 서버 스냅샷
- 위조 customer_id/email 무시
- 고객의 customer_notes 조회 차단 / role→admin 변경 차단
- 공개 스케줄 개인정보 미노출 + 필드 = scheduled_at/duration/label
- 관리자 대시보드·목록·답변 저장(+answered_at)

### 예약(가용성/다중슬롯/취소) E2E (14/14 + 9/9)
- 가용성 응답 개인정보 미포함 / 비로그인 401
- 연속 90분(3슬롯) 예약, duration 저장
- **구간 겹침** 차단(시작·끝·내부 겹침 모두 409)
- 영업시간 초과·최대 슬롯 초과·비정상 duration(45분) 400
- 타인 예약 취소 403 / 본인 취소 200 → cancelled+cancelled_at
- 취소 후 그 슬롯 재예약 가능

### 인증(이메일/비밀번호) (검증)
- 미인증 로그인 차단("Email not confirmed")
- 콜백 token_hash 인증 → 세션 쿠키 + /onboarding
- 인증 후 비밀번호 로그인 성공 / 틀린 비밀번호 거부

> 참고: Supabase는 `@example.com` 등 일부 도메인을 signUp에서 거부함 → 테스트는 admin API로 유저를 만들어 우회.

## 3. 수동 QA 체크리스트

- [ ] 비로그인 `/my`, `/admin` → `/login`
- [ ] 가입 후 이메일 인증 전에는 로그인 불가

- [ ] role 항상 customer / 휴대폰 미입력 시 phone null
- [ ] 프로필 수정에서 name·phone만 변경

- [ ] 관리자가 승인/거절/완료/노쇼 변경 가능

## 4. RLS 직접 검증 (Supabase SQL Editor)

```sql
set role anon;
select * from public.profiles;        -- 0건
select * from public.appointments;    -- 0건
select * from public.customer_notes;  -- 0건
-- 고객 JWT로:
update public.profiles set role='admin' where id=auth.uid();  -- 차단/0건
select * from public.appointments where customer_id != auth.uid(); -- 0건
```
