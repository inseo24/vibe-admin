# 07. 프롬프트 회고 — 현재 코드 기준으로 다시 쓴다면

> 초기에는 "한 덩어리 거대 프롬프트" 하나로 시작했지만, 실제 구현은 여러 번 바뀌었다.
> 이 문서는 **지금 코드 기준으로 프롬프트/문서를 어떻게 구성했어야 했는지** 정리한 회고다.

## 1. 초기 프롬프트 → 현재 코드, 무엇이 달라졌나

| 영역 | 초기 프롬프트 | 현재 코드 | 교훈 |
|------|--------------|-----------|------|
| 로그인 | "이메일 인증 기반 로그인"(매직링크 뉘앙스) | 이메일+비밀번호, 가입 시 인증 1회 | **인증 방식은 처음에 못 박아야** 함 |
| 예약 입력 | datetime 입력 + duration 드롭다운 | 네이버식 캘린더 + 연속 슬롯 다중선택 | UX는 **레퍼런스(네이버) 한 장**이 백 줄보다 명확 |
| 빈 시간 표시 | 명시 없음 | 가용성 API + 잡힌 슬롯 회색 | "남의 예약 안 보이게"는 **명시 안 하면 안 나옴** |
| 예약 취소 | 없음 | 고객 취소 API | 누락 기능. 상태 전이를 처음에 표로 |
| 공개 스케줄 | 리스트 | 월간 캘린더 + 호버 | 표현 형태도 결정 사항 |
| service role | "꼭 필요할 때만" | availability/overlap/cancel/public 4곳 | **어디서 왜 쓰는지** 표로 고정 |
| 배포/운영 | 일반 언급 | 303 로그아웃, vercel.json, SMTP 한계 등 | 운영 디테일은 별도 문서로 |

핵심: **요구사항의 30%가 구현 중 바뀌거나 새로 생겼다.** 거대 단일 프롬프트는 이 변화를 못 따라간다.

## 2. 왜 단일 거대 프롬프트가 문제였나

- 기획·DB·보안·테스트·운영이 **한 곳에 섞여** 있어 무엇이 바뀌면 어디를 고칠지 모호.
- 결정이 안 된 항목(인증 방식, 예약 UX)을 **모호한 문장**으로 남겨 → 구현이 임의로 채움 → 재작업.
- 한 번 쓰고 버려짐. 코드가 진화해도 **프롬프트는 갱신 안 됨** → 문서와 코드 불일치.

## 3. 권장 구성: "단일 프롬프트" 대신 "문서 세트 + 얇은 작업 지시"

이 `docs/` 폴더가 곧 새 구조다. 역할별로 쪼개고, AI에게는 **"이 문서들을 소스 오브 트루스로 삼아 X를 구현하라"**는 얇은 지시만 준다.

```
docs/
  01-기획.md          ← 무엇을/누가/어떤 화면 (PRD)
  02-아키텍처.md      ← 스택/구조/API/설계 결정
  03-데이터베이스.md  ← 스키마/인덱스/RLS
  04-보안-개인정보.md ← 신뢰 경계/권한/개인정보
  05-테스트-QA.md     ← 검증 항목
  06-운영-배포.md     ← env/배포/운영자 설정
```

장점: 변경 시 **해당 문서만** 갱신 → 코드와 문서가 같이 산다. AI 작업도 범위가 좁아 정확.

## 4. 처음에 "결정"으로 못 박았어야 할 것들 (모호하면 재작업)

구현 전에 아래를 **확정 표**로 줬어야 했다:

1. **인증 방식**: 이메일+비밀번호 / 매직링크 / 소셜 중 택1. (← 가장 큰 변경 원인)
2. **예약 모델**: 슬롯 길이, 영업시간, 정원(슬롯당 1), 연속 다중선택 허용 여부, 취소 정책.
3. **상태 전이표**: appointment/contact의 상태와 누가 어떤 전이를 하는지.
4. **공개 노출 범위**: 공개 화면에 무엇을 보이고 무엇을 숨기나(개인정보).
5. **권한 매트릭스**: 역할 × 테이블 × CRUD.
6. **service role 사용처**: 어디서 왜 RLS를 우회하나.

## 5. 재사용 가능한 프롬프트 스켈레톤 (이 프로젝트류)

```
# 역할
너는 Next.js + Supabase + Vercel 어드민을 만드는 시니어 풀스택.

# 소스 오브 트루스
docs/01~06 을 단일 진실로 삼아라. 충돌 시 docs 우선, 코드 다음, 추측 금지.
바뀌는 요구는 먼저 해당 docs를 고친 뒤 구현하라.

# 이번 작업 (얇게)
- 대상: <기능 1개>
- 변경할 docs: <어떤 문서>
- 완료 조건: lint/build 통과 + docs/05 체크리스트 해당 항목 추가·검증

# 불변 제약 (항상)
- 클라이언트 입력(customer_id/email/phone/role/status) 불신, 서버 재구성
- 모든 public 테이블 RLS, service role은 서버 전용+권한검증 후만
- 공개/가용성 API는 시각·상태만 (개인정보 금지)
- 결정 안 된 사항은 임의 구현 말고 질문
```

## 6. 한 줄 결론

> **거대 프롬프트 1개**가 아니라, **역할별 문서 세트(=docs/)** + **결정 표를 먼저 못 박고** + **기능 단위 얇은 지시**로 갔어야 한다. 그러면 요구가 바뀌어도 문서·코드가 함께 진화하고 재작업이 준다.
